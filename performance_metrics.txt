|        |                              |                                                |
|        | Root cause:                  | HIGH_ENTROPY and ENTROPY_TREND signals are      |
|        |                              | mutually exclusive by design - the hard         |
|        |                              | threshold short-circuits before the trend       |
|        |                              | check runs. The test did not account for this   |

|        |                              |                                                |
|        | Fix applied:                 | Replaced trajectory with ["a","a","a","b","c"] |

|        |                              | With entropy_history=[0.5, 0.8], the sequence  |
|        |                              | [0.5, 0.8, 1.371] is strictly increasing and   |
|        |                              | correctly exercises the trend detection path.  |
|        |                              |                                                |
|        | Classification:              | Test logic bug (NOT a production code bug).    |

1. EXTENSION IMPLEMENTATION SUMMARY
To prevent agents from evading exact-argument loop detection by making slightly
varying edits (e.g. progressive sys.path dot appending) or running parallel tool 
calls, we implemented the WorkspaceDensityGuard inside sotis/core/loops.py.

Logic:
  - Tracks file modifications consecutively per target file path.
  - Detects test suite outcomes via result_summary changes.
  - Meltdown triggers if a file is modified >= 3 times without shifting test results.
  - A shift in test results successfully clears all edit counters.
